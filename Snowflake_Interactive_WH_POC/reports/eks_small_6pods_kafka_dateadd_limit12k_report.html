Kafka ingestion ran concurrently with JMeter read load test. The DATEADD(-2sec) query targets freshly ingested Kafka data, proving true real-time read+write on the same Interactive table. The high timeout rate is caused by scanning unoptimized streaming micro-partitions that haven't yet been compacted.
Test Query
SELECT
    CR.CR_ORDER_NUMBER AS ORDER_KEY,
    CR.CR_REFUNDED_CUSTOMER_SK AS CUST_KEY,
    COUNT(DISTINCT CR.CR_REFUNDED_CUSTOMER_SK) OVER (PARTITION BY CR.CR_ORDER_NUMBER) AS ORDER_CUST_COUNT,
    SUM(COALESCE(CR.CR_RETURN_AMOUNT, 0)) AS TOTAL_RETURN_AMOUNT,
    MAX(CR.CR_REASON_SK) AS MAX_REASON,
    MAX(CR.CR_CALL_CENTER_SK) AS MAX_CALL_CENTER,
    MIN(CR.CR_RETURNED_DATE_SK) AS MIN_RETURN_DATE_SK,
    MAX(CR.CR_RETURNED_DATE_SK) AS MAX_RETURN_DATE_SK
FROM INTERACTIVE_DEMO.TPCH_INTERACTIVE.ORDERS_INTERACTIVE AS CR
WHERE CR.INSERTED_AT >= DATEADD('second', -2, CURRENT_TIMESTAMP())
  AND CR.CR_WAREHOUSE_SK = ${__Random(1,25)}
GROUP BY CR.CR_ORDER_NUMBER, CR.CR_REFUNDED_CUSTOMER_SK
LIMIT 12000

Observations & Key Findings

  • 35.1 avg QPS with 40.3% failure rate — significantly lower than the prior static-filter test (342.8 QPS, 0% errors)
  • The DATEADD(-2sec) query targets freshly streamed micro-partitions that are not yet compacted, causing ~759 MB scans vs 14.73 MB in the static-filter test
  • Interactive WH 5-second hard timeout is the bottleneck: successful queries avg 3,378ms server-side, but many exceed the limit
  • Client-side latency is inflated (avg 13,749ms) because JMeter waits for timeout + JDBC error propagation before recording the failure
  • Cluster 2 and 7 show elevated P99 (~20sec) suggesting some queries hit cold micro-partitions from streaming
  • "None" cluster (1,100 queries, avg 7.5s) represents queries that timed out before cluster routing completed
  • Concurrent Kafka streaming at ~26.3 MB/sec (~33.7K msg/s) continuously creates new micro-partitions that the DATEADD query must scan
  • This test proves true concurrent read+write on an Interactive table: queries read data ingested <2 seconds ago while Kafka continuously writes
  • QPS remained stable across all 10 minutes (31.8 – 39.0) with no degradation trend, indicating steady-state behavior
  • To improve success rate: (1) increase WH timeout or use standard WH, (2) reduce DATEADD window to -1 sec, (3) wait for streaming micro-partition compaction
  • Test ran from us-west-2 (Oregon EKS) to Snowflake account in us-west-2
