Test Query
SELECT
    CR.CR_ORDER_NUMBER AS ORDER_KEY,
    CR.CR_REFUNDED_CUSTOMER_SK AS CUST_KEY,
    COUNT(DISTINCT CR.CR_REFUNDED_CUSTOMER_SK) OVER (PARTITION BY CR.CR_ORDER_NUMBER) AS ORDER_CUST_COUNT,
    SUM(COALESCE(CR.CR_RETURN_AMOUNT, 0)) AS TOTAL_RETURN_AMOUNT,
    MAX(CR.CR_REASON_SK) AS MAX_REASON,
    MAX(CR.CR_CALL_CENTER_SK) AS MAX_CALL_CENTER,
    MIN(CR.CR_RETURNED_DATE_SK) AS MIN_RETURN_DATE_SK,
    MAX(CR.CR_RETURNED_DATE_SK) AS MAX_RETURN_DATE_SK
FROM INTERACTIVE_DEMO.TPCH_INTERACTIVE.ORDERS_INTERACTIVE AS CR
WHERE CR.CR_RETURNED_DATE_SK BETWEEN 2452912 AND 2452924
  AND CR.CR_WAREHOUSE_SK IS NOT NULL
GROUP BY CR.CR_ORDER_NUMBER, CR.CR_REFUNDED_CUSTOMER_SK

  • 342.8 avg QPS from JMeter (206,086 queries / 601 seconds) with only 6 pods and 480 threads
  • Peak minute QPS of 472.2 at minute 9 — throughput increased as connection pools warmed
  • Concurrent Kafka streaming at ~23.5 MB/sec had minimal impact on query latency — Snowflake P50 stayed at 168ms

  • Load balanced across all 10 clusters: 9.3% – 11.2% share each
  • Zero errors on both Snowflake-side and JMeter client-side
  • Minutes 1-3 show warmup phase with higher latency (connection pool initialization + SSD cache warming)
  • Minutes 7-10 show steady-state performance with consistent ~450 QPS and sub-1000ms P50
  • Interactive table handled concurrent read (206K queries) + write (49M rows streamed) with no degradation
